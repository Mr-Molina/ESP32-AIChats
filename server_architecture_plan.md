# Backend Server - Architecture Plan

This document outlines the architectural plan for the middle-tier backend server. This server acts as the bridge between the ESP32 atomic audio streamer and the AI engines. It is responsible for handling the persistent WebSocket connection, managing the Model Context Protocol (MCP) tool calls, and orchestrating the STT (Speech-to-Text), LLM (Brain), and TTS (Text-to-Speech) pipeline.

## Finalized Stack Decisions

> [!NOTE]
> **Hosting & Language:** The server will run on a local Debian machine and will be written in **Python**.
> 
> **AI Pipeline Components:**
> 1. **STT (Speech-to-Text):** **Google STT** (Cloud). Offers ultra-low latency streaming recognition. The Python server will pipe the decoded PCM audio directly to Google's API.
> 2. **LLM (Brain):** **Gemini API** (Cloud). Extremely fast (especially `gemini-1.5-flash`) and natively supports Function Calling, which perfectly maps to our MCP JSON-RPC protocol for controlling the ESP32 hardware.
> 3. **TTS (Text-to-Speech):** **Edge-TTS** (Cloud-backed via Python). A brilliant community library that taps into Microsoft's Azure TTS engine for free, providing ultra-realistic voices without requiring API keys.

## Proposed Architecture (Python / FastAPI)

The server will be built using Python, leveraging `FastAPI` to handle WebSocket connections asynchronously.

### 1. Connection & Protocol Layer
*   **WebSocket Handler:** Accepts incoming connections from the ESP32. Maintains a dictionary of active sessions (allowing multiple ESP32s to connect simultaneously).
*   **MCP Protocol Manager:** Parses incoming JSON-RPC 2.0 messages. It routes binary Opus audio payloads to the Audio Transcoder and handles `tools/call` responses.

### 2. The Audio Transcoder Layer
The ESP32 sends and expects compressed Opus frames to save bandwidth, but AI models require raw PCM audio or text.
*   **Opus Decoder:** Uses `opuslib` to decompress the incoming 20ms/40ms Opus frames from the ESP32 into raw 16kHz PCM data.
*   **Opus Encoder:** Compresses the raw PCM audio generated by the TTS engine back into Opus frames before streaming them down the WebSocket.

### 3. The AI Orchestration Pipeline
This is the core loop that runs when the ESP32 signals that the user has stopped speaking (VAD End-Of-Speech).

1.  **STT (Speech-to-Text):** The accumulated PCM audio buffer from the user is passed to the STT engine. It returns a transcribed text string.
2.  **LLM (Language Model):** The text is injected into the conversation history and sent to the LLM. 
    *   *Tool Calling:* The LLM is given access to tools (e.g., "turn_on_led"). If the LLM decides to use a tool, the server translates this into an MCP JSON-RPC message and fires it down the WebSocket to the ESP32 immediately.
3.  **TTS (Text-to-Speech):** The LLM's text response is sent to the TTS engine, which generates a raw PCM audio stream.
4.  **Streaming Output:** As the TTS engine generates audio chunks, they are immediately passed to the Opus Encoder and streamed down the WebSocket to the ESP32, ensuring minimal conversational delay.

## Proposed Directory Structure

```text
atomic-backend-server/
├── requirements.txt
├── .env                        # API keys and configuration
├── main.py                     # FastAPI app and WebSocket endpoints
├── core/
│   ├── connection_manager.py   # Handles active WS sessions
│   ├── mcp_protocol.py         # JSON-RPC parser and formatter
│   └── audio_transcoder.py     # Opus encode/decode wrappers
└── ai_pipeline/
    ├── orchestrator.py         # Manages the STT -> LLM -> TTS flow
    ├── stt_engine.py           # Interface for Whisper or Cloud STT
    ├── llm_engine.py           # Interface for Ollama or Gemini/OpenAI
    └── tts_engine.py           # Interface for Piper or Cloud TTS
```

## Verification Plan

### 1. Local Echo Test
Before connecting any AI, build an "echo" mode. The server receives Opus frames, decodes them, immediately re-encodes them, and sends them back. This verifies the WebSocket stability and the `opuslib` integration.

### 2. Tool Call Test
Trigger a simulated tool call from the server (e.g., `{"method": "tools/call", "params": {"name": "led_color", "arguments": {"color": "red"}}}`). Verify the ESP32 parses the JSON-RPC and changes the LED color without crashing the audio stream.

### 3. Full Pipeline Test
Connect the ESP32 and speak a prompt. Measure the "Time to First Byte" (TTFB)—the latency between the user finishing their sentence and the ESP32 playing the first Opus frame of the AI's response.
